1. Abstract

Migration Reporting Tool is a web-based utility. Designed as a solution for visualizing, editing, and reporting migration data processes, the tool integrates user observations and data from various sensors to generate, in real-time, an interactive map depicting migrations conducted by different entities.

2. Introduction

Migration Reporting Tool (MIR) is a project proposed at Web Application Development course. In order to resolve this task, the REST paradigm will be used. The communication between the server and the clients will be resolved by creating an REST API that will handle the desired comunnication in our web application.The migrations will be shown on an interactive map and they will be handled in different ways by the users, regarding the role of the client.

3. Motivation

We opt for this project because it is an interisting problem in real life, regarding some situations: firstly, such a tool will be useful to see how Romanians go abroad because of the financial situation. Then, we taught about the migrations that took place because of the wars. Finally, we can see the impact of such an application when we want to track how birds or some species of animals migrates because of the weather/ climate changes.

4. Functionalities

4.1 Internal Data

The internal data structures and models chosen for MIR will be designed to efficiently store and manage migration data. In this context, we think about the following models:

  • Migration Object Model
  • User Role-Based Model

In the context diagram, we want to show this type of hadling. In the below diagram, we show that we will divide the data into two main components: users and migrations (see the databases).

In the containers diagram, we focused on the interaction of clients, regarding their status (user or guest), in order to show the desired behaviour for our web application.

As seen in the diagrams above we'll be having the following functionalities an user can interact with throughout our website:

    1. Login
    2. Register
    3. Create a new migration
    4. Update a migration
    5. Delete a migration
    6. See a migration in details
    7. See a list of migrations
    8. Interact with an inteligent map that displays migrations

In the early stage of building the architecture with MIR, we have thought that the following models will suit perfectly the needs for our models:

    1. User
    2. Migration
    3. MigrationType
    4. MigrationContext
    5. MigrationLocation

4.2. External Data Sources

MIR will use external data sources such as:

  • Migration Data API - that will help us in managing the functionalities
  • Geolocation Services - that will help us in showing the migrations on a map
    • The interactive map will be implemented using Mapbox since it fits all criterias we are looking for.

4.3 Conforming with linked data principles

The solution implemented in MIR will use linked data principles, ensuring data interoperability and accessibility. For our project, we think at these possible solutions:

  • Usage of Uniform Resource Identifiers (URIs) for data identification
  • Linking data elements through RDF (Resource Description Framework)

4.4 Functionalities in depth

This section will describe the functionalities mentioned in 4.1
    1. Login
      2. A guest can log in to become a user of the application in order to have access to some functionalities as: create a migration, delete a migration, edit a migration, see the list of migrations a user has created.
    2. Register
      3. A guest can register to become a user of the application in order to have access to some of the functionalities as: create a migration, delete a migration, edit a migration, see the list of migrations a user has created. A guest will be restrictioned from accessing every functionality on the website, but he will have access to: interaction with map, seeing one migration in detail.
    3. CRUD on migration
      4. This can only be performed by a user throughout our website. They will be able to create, read, update and delete a migration interacting with our interface.
    4. See a list of migrations
      5. This can only be performed by a user. This functionality will allow the user to see all the migrations he has created giving giving him the posibility of editing and deleting the migration.
    5. Interaction with inteligent map
      6. Either an user or a guest can have access to this functionality. They will be able to interact with this map to see what others have posted.
